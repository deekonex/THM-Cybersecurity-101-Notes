# 5.03 Networking — Core Protocols (Concise)

Overview

This note summarizes essential Application Layer protocols used for web, email, file transfer, and name resolution. Focus is on purpose, default ports, key commands/records, security notes, and quick examples.

Quick ports

- DNS: UDP/TCP 53 (UDP for queries; TCP for zone transfer and large responses)
- HTTP: TCP 80
- HTTPS: TCP 443 (TLS)
- FTP (control): TCP 21; (data) TCP 20 (active) or ephemeral ports (passive)
- SFTP (SSH File Transfer): TCP 22 (not FTP over SSH)
- SMTP (mail transfer): TCP 25; Submission: TCP 587 (with STARTTLS); SMTPS (implicit TLS): TCP 465
- POP3: TCP 110; POP3S (TLS): TCP 995
- IMAP: TCP 143; IMAPS (TLS): TCP 993
- WHOIS: TCP 43 (query to registrar)

DNS (Domain Name System)

- Purpose: Resolve domain names → IP addresses and provide service metadata.
- Common record types: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail exchange), NS (name server), TXT (text/SPF/DKIM), PTR (reverse DNS), SOA (zone authority), SRV (service locator).
- Security: DNS is unauthenticated by default; DNSSEC adds signature-based integrity and origin validation.
- Tools/examples:
  - dig example.com A
  - nslookup -type=MX example.com
- Note: Responses typically use UDP 53; TCP 53 is used for zone transfers (AXFR) and when responses exceed UDP size limits.

WHOIS

- Purpose: Lookup domain registration metadata (registrant, registrar, creation/expiry). It queries WHOIS servers (TCP 43) and registrar databases.
- Limitations: Many fields are redacted due to privacy/GDPR; accuracy depends on registrar data.
- Tool: whois example.com

HTTP / HTTPS

- Purpose: Transfer hypertext and APIs (request/response model).
- Important methods: GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS.
- Common headers: Host, User-Agent, Accept, Content-Type, Authorization, Cookie, Set-Cookie, Location (redirect).
- Status codes: 1xx (informational), 2xx (success), 3xx (redirect), 4xx (client error), 5xx (server error).
- HTTPS: HTTP over TLS. TLS provides confidentiality, integrity and server authentication via certificates. Modern practices:
  - Use TLS 1.2+; prefer 1.3
  - Validate cert chains and check hostname/SNI
  - Use HSTS, secure cookies, and TLS ciphersuites with forward secrecy
- Quick examples:
  - curl -I https://example.com      # show response headers
  - curl -o page.html http://10.10.10.5/flag.html
  - Manual raw request (telnet/netcat to port 80):
    GET /path HTTP/1.1\r\nHost: example.com\r\n\r\n

FTP / SFTP

- FTP (legacy): control on TCP 21, data via separate connection. Modes: active (server connects back) and passive (client connects to server-specified port). Credentials may be sent plaintext unless FTPS is used.
- FTPS: FTP over TLS (explicit via AUTH TLS or implicit on different port).
- SFTP: File transfer over SSH (port 22). Not related to FTP protocol—preferred secure option.
- Common commands: USER, PASS, LIST, RETR (download), STOR (upload), TYPE (A/B), PASV.
- Example (anonymous):
  ftp 10.10.10.5
  # username: anonymous

SMTP (sending email)

- Purpose: Server-to-server mail transfer; clients submit mail for delivery.
- Ports: 25 (MTA to MTA), 587 (submission with STARTTLS + auth), 465 (legacy implicit TLS submission).
- Key commands: HELO/EHLO, MAIL FROM:, RCPT TO:, DATA, QUIT, STARTTLS.
- Interaction with DNS: Mail routing uses MX records pointing to mail servers.
- Security: Use STARTTLS or implicit TLS and authentication. Email often traverses multiple MTAs; rely on SPF/DKIM/DMARC for sender validation.

POP3 vs IMAP

- POP3 (Post Office Protocol v3): Download-and-optional-delete model. Typical use: single-device download. Plaintext on 110; POP3S on 995.
  - Commands: USER, PASS, STAT, LIST, RETR, DELE, QUIT.
- IMAP (Internet Message Access Protocol): Synchronizing model; messages stay on server, support folders, flags, partial fetch, move.
  - Ports: 143 (IMAP), 993 (IMAPS TLS).
  - Commands: LOGIN, SELECT, FETCH, STORE, COPY, MOVE, LOGOUT.
- Security note: Plain-text protocols expose credentials—use TLS (STARTTLS or implicit TLS) and prefer modern auth mechanisms.

Packet capture observation

- Plaintext protocols reveal credentials and content (e.g., HTTP non-TLS, FTP, SMTP without STARTTLS, POP3). Use TLS-protected variants to hide sensitive data.
- Use tshark/wireshark: tshark -r capture.pcap -Y "http or ftp or smtp" -V

Practical tips & checks

- Prefer secure alternatives: HTTPS, SFTP, FTPS (or better, SFTP), IMAPS/POP3S, SMTP submission over 587 with STARTTLS.
- For email deliverability/trust: configure SPF (TXT), DKIM (DNS TXT signatures), and DMARC (DNS TXT policy).
- For DNS troubleshooting: use dig +short, dig @8.8.8.8 example.com ANY, and check TTLs, NS records and zone delegation.
- When testing services on VMs or CTF boxes: check service ports with nmap (-sV -sT/-sU), and try curl/wget, telnet/nc, and appropriate client tools.
