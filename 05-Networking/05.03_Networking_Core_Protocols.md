# 5.03 Networking — Core Protocols

Overview

This note explains the common Application Layer protocols used for name resolution, web access, file transfer, and email. For each protocol you'll find the purpose, default ports, basic commands/records, security notes, and quick commands to try. It's written for beginners and focuses on the practical essentials.

Quick ports (default)

- DNS: UDP/TCP 53 (UDP for most queries; TCP for zone transfer and large responses)
- HTTP: TCP 80
- HTTPS: TCP 443 (HTTP over TLS)
- FTP (control): TCP 21; data: TCP 20 (active) or ephemeral ports (passive)
- SFTP (SSH File Transfer): TCP 22 (this is SSH-based file transfer, not "FTP over SSH")
- FTPS (FTP over TLS): usually TCP 21 (explicit) or other ports (implicit FTPS historically used 990)
- SMTP (mail transfer): TCP 25 (MTA to MTA)
- SMTP submission: TCP 587 (client submission, STARTTLS recommended)
- SMTPS (legacy implicit TLS): TCP 465
- POP3: TCP 110; POP3S (TLS) TCP 995
- IMAP: TCP 143; IMAPS (TLS) TCP 993
- WHOIS: TCP 43 (querying registrar WHOIS servers)

DNS (Domain Name System)

- Purpose: Translate human-friendly domain names (example.com) into IP addresses and provide service metadata.
- Common record types (simple explanations):
  - A: IPv4 address for a name.
  - AAAA: IPv6 address for a name.
  - CNAME: Alias from one name to another (useful for www → root).
  - MX: Mail Exchange — which server accepts email for the domain.
  - NS: Authoritative name servers for the domain.
  - TXT: Arbitrary text (commonly used for SPF, DKIM, DMARC verification).
  - PTR: Reverse DNS mapping from IP → name.
  - SOA: Start of Authority — zone metadata (serial, refresh, admin contact).
  - SRV: Service locator records (used by some apps/protocols).
- Security: By default DNS responses are unauthenticated. DNSSEC adds signatures so resolvers can verify integrity and origin.
- Quick tools/examples:
  - dig example.com A
  - dig +short example.com
  - dig @8.8.8.8 example.com ANY
  - nslookup -type=MX example.com

WHOIS (domain registration lookup)

- Purpose: Query domain registration metadata (registrant, registrar, creation/expiry dates).
- Notes and limitations:
  - WHOIS data comes from registrars and registries; accuracy depends on what was provided.
  - Many records are redacted or replaced by privacy services (GDPR/privacy masks).
  - For programmatic/modern lookups, use registrar APIs or RDAP where available.
- Quick command: whois example.com

HTTP / HTTPS (web)

- Purpose: Transfer web pages, APIs and resources using a request/response model.
- Important methods: GET (read), POST (create/submit), PUT (replace), PATCH (modify), DELETE, HEAD, OPTIONS.
- Common headers you'll see: Host, User-Agent, Accept, Content-Type, Authorization, Cookie, Set-Cookie, Location.
- Status codes (short): 1xx informational, 2xx success, 3xx redirect, 4xx client error, 5xx server error.
- HTTPS: HTTP over TLS. TLS provides encryption, integrity, and server authentication via certificates.
  - Use TLS 1.2+ (prefer TLS 1.3).
  - Validate certificates (chain and hostname). Use HSTS and secure cookie flags where appropriate.
- Try these examples:
  - curl -I https://example.com   # show only headers
  - curl -o page.html http://10.10.10.5/flag.html
  - Raw request with netcat/telnet: connect to port 80 and send a minimal request:
    GET /path HTTP/1.1\r\nHost: example.com\r\n\r\n

FTP, FTPS, SFTP (file transfer)

- FTP: An older protocol for file transfer. Control connection on TCP 21; data transfers use a separate connection and different ports depending on active/passive mode.
  - Modes: Active (server connects back to client for data) and Passive (client connects to server-specified port).
  - Commands: USER, PASS, LIST, RETR, STOR, TYPE, PASV, PORT.
  - Security: FTP sends credentials and data in plaintext unless wrapped by TLS (FTPS) — avoid using plain FTP on public networks.
- FTPS: FTP over TLS. Can be explicit (AUTH TLS on port 21) or implicit (historical convention, other port).
- SFTP: File transfer over SSH (port 22). Different protocol from FTP; SFTP is secure and commonly preferred over FTP/FTPS for simplicity and firewall friendliness.

SMTP (sending email)

- Purpose: Transfer email between mail servers and accept submissions from mail clients.
- Ports and purpose:
  - 25: Server-to-server (MTA to MTA).
  - 587: Submission port for mail clients; STARTTLS + authentication expected.
  - 465: Legacy implicit TLS (still used by some providers).
- Basic SMTP commands: HELO/EHLO, MAIL FROM:, RCPT TO:, DATA, . (single dot on a line ends message), QUIT, STARTTLS.
- Email authentication and trust:
  - SPF (DNS TXT) — which hosts can send mail for the domain.
  - DKIM (DNS TXT) — cryptographic signatures for outgoing mail.
  - DMARC (DNS TXT) — policy combining SPF/DKIM for enforcement/reporting.

POP3 vs IMAP (receiving email)

- POP3 (Post Office Protocol v3): Simple model — client downloads messages (optionally deletes them from server). Good for a single device.
  - Ports: 110 (plaintext), 995 (POP3S, TLS).
  - Commands: USER, PASS, STAT, LIST, RETR <id>, DELE <id>, QUIT.
- IMAP (Internet Message Access Protocol): Keeps messages on the server and supports folders, flags and selective fetches. Good for multiple devices.
  - Ports: 143 (plaintext/STARTTLS), 993 (IMAPS, implicit TLS).
  - Common commands: LOGIN, SELECT <mailbox>, FETCH <msg>, STORE, MOVE, LOGOUT.
- Security note: Plaintext logins expose credentials — prefer TLS (STARTTLS or implicit TLS) and modern auth (OAuth where supported).

Observing protocols with packet capture

- Plaintext protocols (HTTP without TLS, FTP, POP3, SMTP without STARTTLS, IMAP without TLS) reveal credentials and message contents to an eavesdropper.
- Use Wireshark/tshark for inspection. Example:
  - tshark -r capture.pcap -Y "http or ftp or smtp" -V
- When you see credentials or flags in captures, that shows why TLS is important.

Practical tips & checks

- Prefer secure variants: HTTPS, SFTP, IMAPS/POP3S, SMTP submission over 587 with STARTTLS.
- For email deliverability and anti-spoofing: configure SPF, DKIM, and DMARC in DNS.
- DNS troubleshooting: dig +short, dig @8.8.8.8 example.com ANY, check TTLs and NS delegation.
- Service troubleshooting on VMs/CTFs: nmap -sS -sV -p- <target>, curl/wget, telnet/nc, ftp/sftp clients.

Commands reference

- dig example.com A
- nslookup -type=MX example.com
- whois example.com
- curl -I https://example.com
- wget http://10.10.10.5/flag.html
- ftp 10.10.10.5
- sftp user@10.10.10.5
- telnet <host> 80 (HTTP) / 25 (SMTP) / 110 (POP3) / 143 (IMAP)
- nmap -sS -sV -p- 10.10.10.5

Takeaways

- Ports are conventions — services _usually_ listen on standard ports but can run anywhere.
- "SFTP" is not "FTP over SSH" — it's a distinct protocol that runs over SSH.
- "FTPS" is FTP wrapped in TLS; it can be confusing because FTP itself uses separate data channels.
- Use telnet/netcat only for learning and debugging text-based protocols; avoid sending real credentials over plaintext connections on public networks.
